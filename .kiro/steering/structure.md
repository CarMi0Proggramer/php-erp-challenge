# Project Structure

## Backend (`app/`)

```
app/
├── Actions/Fortify/        # Fortify action overrides (e.g., password reset)
├── Concerns/               # Reusable validation rule traits (used in FormRequests)
├── Enums/                  # PHP backed enums: Role, ProductSize, StockMovementType
├── Http/
│   ├── Controllers/        # Grouped by domain (Products/, Settings/)
│   ├── Middleware/         # EnsureUserHasRole, HandleInertiaRequests, HandleAppearance
│   └── Requests/           # FormRequests grouped by domain, use Concern traits for rules
├── Jobs/                   # Queued jobs (e.g., DeleteProductImagesJob)
├── ModelFilters/           # EloquentFilter classes, one per filterable model
├── Models/                 # Eloquent models; use #[Fillable] and #[Hidden] PHP attributes
├── Observers/              # Model observers registered via #[ObservedBy] on the model
├── Policies/               # One policy per model; Admin bypass via before()
├── Providers/              # AppServiceProvider (configures defaults), FortifyServiceProvider
└── Services/               # Business logic services (e.g., StockMovementService)
```

## Routes (`routes/`)

- `web.php` — entry point, requires domain-specific route files
- `products.php`, `users.php`, `settings.php` — domain route files
- All routes require `auth` + `verified` middleware; role restrictions use `role:` middleware alias

## Database (`database/`)

- `migrations/` — standard Laravel migrations
- `factories/` — one factory per model
- `seeders/` — `DatabaseSeeder` calls domain seeders (e.g., `UserSeeder`)
- Models use UUIDs (`HasUuids` trait) as primary keys

## Frontend (`resources/js/`)

```
resources/js/
├── app.ts                  # Inertia app bootstrap; layout assigned by page name prefix
├── components/
│   ├── ui/                 # Generic UI primitives (Button, Table, Select, etc.) built on reka-ui
│   ├── products/           # Domain-specific components
│   ├── users/              # Domain-specific components
│   └── wysiwyg/            # Tiptap editor wrapper
├── composables/            # Vue composables (useAppearance, useTwoFactorAuth, etc.)
├── layouts/                # AppLayout, AuthLayout, SettingsLayout
├── lib/
│   ├── enums/              # TypeScript enums mirroring PHP enums
│   └── utils/              # cn(), debounce(), formatDate(), etc.
├── pages/                  # Inertia page components, grouped by domain
│   ├── auth/
│   ├── products/
│   └── settings/
├── routes/                 # Auto-generated by Wayfinder — DO NOT edit manually
├── actions/                # Auto-generated by Wayfinder — DO NOT edit manually
├── types/                  # TypeScript types; domain types in separate files, re-exported from index.ts
└── wayfinder/              # Wayfinder runtime helpers — DO NOT edit manually
```

## Tests (`tests/`)

```
tests/
├── Concerns/               # Shared test traits (e.g., HasProductRoleProviders for DataProviders)
├── Feature/                # HTTP/integration tests grouped by domain
└── Unit/                   # Unit tests for Jobs, Observers, etc.
```

## Key Conventions

- **Layout routing**: Inertia layout is determined by page name prefix in `app.ts` (`auth/` → AuthLayout, `settings/` → nested SettingsLayout, default → AppLayout). Pages can also define `layout.breadcrumbs` via `defineOptions`.
- **Authorization**: Use `#[Authorize]` attribute on controller methods or `$this->authorize()`. Policies always have a `before()` that grants Admin full access.
- **Validation**: Validation rules live in `Concerns/` traits and are composed into `FormRequest` classes.
- **Filtering**: Query filters are handled by `ModelFilter` classes in `app/ModelFilters/`. Add a new method per filterable query param.
- **Flash messages**: Shared via Inertia middleware as `flash.message` (success) and `flash.error` (error).
- **Wayfinder routes**: Import from `@/routes` or `@/routes/{domain}` in Vue components. Never construct URLs manually.
- **Types**: All shared TypeScript types go in `resources/js/types/`. Add new domain type files and re-export from `index.ts`.

# Architecture

- Use modular architecture
- Separate business logic from framework code
- Avoid fat controllers and fat models
- Prefer services for business logic
- Keep domain rules centralized

## Backend Structure

- Controllers → only handle HTTP
- Services → business logic
- Models → data representation and relationships
- Policies → authorization

## Rules

- No business logic in controllers
- No direct DB queries in controllers
- Keep functions small and focused
